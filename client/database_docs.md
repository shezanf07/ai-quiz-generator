## 1. Database Overview

The database layer is owned by Person 3 in the team split. It uses MongoDB 6.0+ hosted on MongoDB Atlas (free M0 tier) and is accessed exclusively through Mongoose ODM from the backend services. The database is named `quiz_generator` and contains 5 collections covering all data needs of the application.

> **Why MongoDB?**
> MongoDB's flexible JSON-like document model is ideal for this project. Quiz questions have variable structures (MCQ, True/False, Fill-in-blank). The embedded `options[]` array in each question is a natural fit for document storage. Horizontal scalability and native Node.js integration via Mongoose make it the right choice for a MERN stack app.

### 1.1 Collections at a glance

| Collection | Documents store | Owned by module | Approx. doc size |
| :--- | :--- | :--- | :--- |
| **users** | Educator accounts, auth provider, preferences, settings | Auth | ~1 KB per user |
| **quizzes** | Quiz metadata, settings, link data, analytics summary | Quiz Management | ~3 KB per quiz |

| **results** | Student session, answers array, score, time, device info | Scoring | ~2 KB per result |
| **templates** | CSS theme configs for the 5 UI themes (seeded at startup) | Admin / seeded | ~1 KB per template |

## 3. Relationships (ERD)

MongoDB is non-relational, but Mongoose ObjectId references create logical relationships between collections. The diagram below shows the full relationship chain. References (FK) are used instead of embedding for collections that need to be independently queried, updated, or have unbounded growth potential.

| From | To | Type | Field | Reason for referencing vs embedding |
| :--- | :--- | :--- | :--- | :--- |
| quizzes | users | Many → One | `creator_id` | One user creates many quizzes. User data changes independently. |
| quizzes | questions | One → Many | `questions[ ]` | Questions can be individually edited, reordered, regenerated without touching the quiz doc. |
| quizzes | templates | Many → One | `template_id` | Multiple quizzes can share one template. Templates managed separately. |
| results | quizzes | Many → One | `quiz_id` | One quiz can have many result docs. Results grow unboundedly. |
| results | users | Many → One* | `participant_info.user_id` | Optional - anonymous students don't have a user account. |
| results | questions | Embedded ref | `answers[].question_id` | Each answer in the answers array references the question it answers. |

### 3.1 Why references over embedding?

A common MongoDB decision is whether to embed sub-documents or reference them. Here is the reasoning for each key decision in this project:

* **questions are referenced** (not embedded in quizzes) — a quiz can have 50+ questions. Embedding would make the quiz document very large and every edit to one question would require rewriting the whole quiz document.
* **results are referenced** (not embedded in quizzes) — results grow with every student attempt and could be thousands of documents. Embedding would make the quiz document grow unboundedly.
* **options[] ARE embedded** inside questions — each question has exactly 4 options that only exist as part of that question. Embedding is correct here.
* **answers[] ARE embedded** inside results — each result owns its answers. They are never queried independently, so embedding is correct.
* **templates are referenced** — multiple quizzes share templates. Embedding would duplicate the CSS config in every quiz document.

## 4. Indexes and Performance

Indexes are one of Person 3's primary responsibilities. Without the right indexes, queries that run on every page load will do full collection scans, making the app unacceptably slow as data grows. Each index below is tied to a specific, frequently-used query.

### users indexes
| Index definition | Options | Purpose |
| :--- | :--- | :--- |
| `{ email: 1 }` | **unique: true** | Login lookup by email — runs on every login request. Unique enforces one account per email. |
| `{ created_at: -1 }` | | Admin user list sorted by newest — descending sort without full scan. |

### quizzes indexes
| Index definition | Options | Purpose |
| :--- | :--- | :--- |
| `{ creator_id: 1, created_at: -1 }` | **compound** | Creator dashboard — fetch all quizzes by a user, sorted newest first. Most frequent creator-side query. |
| `{ "link.short_id": 1 }` | **unique: true** | Student quiz access via short link — runs on every student page load. Must be fast. |
| `{ status: 1 }` | | Filter quizzes by status (published / draft / archived) for admin views. |

### questions indexes
| Index definition | Options | Purpose |
| :--- | :--- | :--- |
| `{ quiz_id: 1, display_order: 1 }` | **compound** | Fetch all questions for a quiz in the correct display order — runs when quiz loads for student. |
| `{ difficulty: 1 }` | | Filter or sort questions by difficulty level for analytics and question bank features. |

### results indexes
| Index definition | Options | Purpose |
| :--- | :--- | :--- |
| `{ quiz_id: 1, created_at: -1 }` | **compound** | Creator results list — all results for a quiz, sorted by newest. Used on analytics dashboard. |
| `{ "participant_info.user_id": 1 }` | | Fetch all results for a specific registered student across quizzes. |
| `{ completed_at: 1 }` | | TTL index candidate — expire abandoned sessions after 24 hours. |

### templates indexes
| Index definition | Options | Purpose |
| :--- | :--- | :--- |
| `{ category: 1 }` | | Filter templates by theme category (dark, minimal, colorful, etc). |
| `{ is_public: 1 }` | | Fetch only public templates for the creator theme picker. |

> **Performance tip:** The two most critical indexes are `{ link.short_id }` unique on quizzes and `{ quiz_id, display_order }` compound on questions. These run on every student quiz page load. Without them, both queries do a full collection scan and will become the primary bottleneck as quiz and question counts grow.

## 5. Sample Queries

These are the 5 most important Mongoose queries Person 3 will write and optimize. Each is tied directly to a key application feature and uses the indexes defined above.

### Query 1 — Creator dashboard: get all quizzes by a creator
```javascript
// Uses compound index: { creator_id: 1, created_at: -1 }
const quizzes = await Quiz
  .find({ creator_id: userId })
  .sort({ created_at: -1 })
  .populate('template_id', 'name category')
  .lean();
```

### Query 2 — Student access: get quiz by short link
```javascript
// Uses unique index: { link.short_id: 1 }
const quiz = await Quiz
  .findOne({ 'link.short_id': shortId, status: 'published' })
  .populate('template_id')
  .lean();

if (!quiz || (quiz.expires_at && quiz.expires_at < new Date()))
  throw new Error('Quiz not found or expired');
```

### Query 3 — Load questions for a quiz in order
```javascript
// Uses compound index: { quiz_id: 1, display_order: 1 }
const questions = await Question
  .find({ quiz_id: quizId })
  .sort({ display_order: 1 })
  .select('-manual_review') // exclude internal review notes
  .lean();
```

### Query 4 — Save a completed result
```javascript
const result = await Result.create({
  quiz_id: quizId,
  participant_info: { name, email, ip_address: req.ip },
  score: correctCount,
  percentage: (correctCount / total) * 100,
  passed: percentage >= quiz.settings.passing_score,
  answers: sessionAnswers,
  total_time_taken: timeTaken,
  started_at: session.startTime,
  completed_at: new Date(),
  status: 'completed',
});
```

### Query 5 — Creator analytics: aggregate results for a quiz
```javascript
// MongoDB aggregation pipeline - no index needed (uses quiz_id index)
const analytics = await Result.aggregate([
  { $match: { quiz_id: mongoose.Types.ObjectId(quizId),
    status: 'completed' } },
  { $group: {
    _id: '$quiz_id',
    total: { $sum: 1 },
    avg_score: { $avg: '$percentage' },
    pass_rate: { $avg: { $cond: ['$passed', 1, 0] } },
    avg_time: { $avg: '$total_time_taken' }
  }}
]);
```

## 6. Atlas Setup and Deployment

MongoDB Atlas is the managed cloud hosting platform for the database. Person 3 is responsible for the full Atlas setup. The M0 free tier provides 512 MB storage, shared RAM, and automated backups - more than enough for a BCA final year project.

### 6.1 Step-by-step Atlas setup

1. Go to cloud.mongodb.com and create a free account
2. Create a new Project named quiz-generator
3. Build a free M0 cluster - choose AWS, region closest to your location (e.g. Mumbai ap-south-1)
4. Under Security > Database Access, create a DB user with username `quizapp` and a strong password. Grant readWrite role on `quiz_generator` database only
5. Under Security > Network Access, add IP address `0.0.0.0/0` during development. For production add only the Render.com backend server IP
6. Click Connect on the cluster, choose Connect your application, copy the connection string
7. Replace `<password>` in the string with your DB user password. Add `/quiz_generator` as the database name before the `?`
8. Paste the full URI into your `.env` file as `MONGODB_URI`
9. In `config/db.js`, connect with `mongoose.connect(process.env.MONGODB_URI, { dbName: 'quiz_generator' })`
10. Run `node scripts/seedTemplates.js` once to insert the 5 template documents into the templates collection

### 6.2 Mongoose connection config (`config/db.js`)
```javascript
const mongoose = require('mongoose');

const connectDB = async () => {
  try {
    await mongoose.connect(process.env.MONGODB_URI, {
      dbName: 'quiz_generator',
      serverSelectionTimeoutMS: 5000,
      socketTimeoutMS: 45000,
    });
    console.log('MongoDB Atlas connected');
  } catch (err) {
    console.error('DB connection failed:', err.message);
    process.exit(1);
  }
};

module.exports = connectDB;
```

### 6.3 Atlas features to enable

| Feature | Where to enable | Why |
| :--- | :--- | :--- |
| Automated backups | Atlas Cluster > Backup | Daily snapshots - restore if data is accidentally deleted |
| Performance Advisor | Atlas Cluster > Performance | Atlas suggests missing indexes based on slow query logs |
| Real-time monitoring | Atlas Cluster > Metrics | CPU, memory, connections, query latency dashboards |
| Database triggers | Atlas App Services > Triggers | Optional - run functions on document insert/update events |
| Search indexes | Atlas Search | Optional - full-text search across quiz titles and questions |
